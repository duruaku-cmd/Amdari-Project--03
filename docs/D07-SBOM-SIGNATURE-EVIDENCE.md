# D-07 — SBOM & Signature Evidence

This deliverable proves SentinelPay container images are signed and accompanied by
a signed software bill of materials, and that deployment verification fails closed.

## What is produced on every build

1. **Container image** built from `services/payments-api` in CI.
2. **Trivy image scan** — blocks the build on CRITICAL/HIGH (fixable) OS or app
   CVEs. Scanning the *image* (not just dependencies) covers the base-image layers.
3. **CycloneDX SBOM** (`sbom.cdx.json`) generated by Syft and uploaded as a build
   artefact — closes V-PIP-02. This is the machine-readable record of every
   component in the image.
4. **Cosign keyless signature** over the image — closes V-PIP-01. Signing uses the
   workflow's GitHub OIDC identity to obtain a short-lived Fulcio certificate; the
   signature is recorded in the Rekor transparency log. No long-lived signing key
   exists to be stolen.
5. **Signed SBOM** attached to the image as an OCI artefact.

## Verification (fails closed) — scripts/verify-image.sh

Before any deployment, `verify-image.sh <image>` runs `cosign verify`, pinned to
the **expected GitHub identity and OIDC issuer**. It exits non-zero (blocking
deploy) if the image is:

- unsigned,
- tampered with after signing,
- signed by an unexpected identity (e.g. an attacker), or
- missing its signed SBOM.

Verified locally: the script returns exit 1 on a failed/absent signature and exit
0 only when both the image signature and the signed SBOM validate.

## What this protects against — and what it does not (brief Q4)

**Protects against:** a registry-tampering or image-substitution attack. If an
attacker replaces the image in ECR, their image is not signed by our identity, so
verification fails closed and deployment is blocked. It also gives provenance: the
signature proves which workflow built the image.

**Does not protect against:** a compromise of the *build itself*. If an attacker
injects malicious code into the source or the build pipeline before signing, the
resulting image is "validly" signed — signing attests provenance, not innocence.
That residual risk is mitigated by the other gates (secret scan, SAST, SCA, IaC,
image scan) and by branch protection requiring review before merge.
